What is Amazon DynamoDB?

  • Consistent performance

DynamoDB Core Characteristics

Serverless

Zero infrastructure management, no maintenance windows, automatic patching

NoSQL

Key-value and document data models, flexible schema, no JOIN operations

Fully Managed

AWS handles scaling, monitoring, security, backups automatically

High Performance

Single-digit millisecond latency, consistent at any scale

Key Difference from Relational Databases: Purpose-built for scale and performance, trades query flexibility for speed and scalability

Core Components & Terminology

Tables

Collection of data items. Schemaless except for primary key requirement.

Items

Individual records (like rows). Each item can have different attributes.

Attributes

Basic data elements (like columns). Support various data types.

Data Types

  • Scalar: String, Number, Binary, Boolean, Null
  • Document: List, Map
  • Set: String Set, Number Set, Binary Set

Primary Keys - The Foundation

Simple Primary Key

Partition Key Only

  • Uniquely identifies each item
  • Good for simple lookups
  • Example: User profiles
Table: Users Primary Key: UserId (Partition Key)

Composite Primary Key

Partition Key + Sort Key

  • Enables range queries
  • Multiple items per partition
  • Example: Time-series data
Table: GameScores Partition Key: UserId Sort Key: GameTitle
Critical Design Decision: Primary key structure determines query patterns and performance. Choose high-cardinality partition keys for even distribution.

Basic Operations

Item Operations

  • PutItem: Create/replace item
  • GetItem: Retrieve by primary key
  • UpdateItem: Modify attributes
  • DeleteItem: Remove item

Batch Operations

  • BatchGetItem: Up to 100 items
  • BatchWriteItem: Up to 25 items

Query vs Scan

Query Scan
Uses primary key Examines every item
Efficient & fast Less efficient
Limited capacity consumption Consumes full table capacity
Preferred method Use sparingly
8

Secondary Indexes

Global Secondary Index (GSI)

  • Different partition and sort keys
  • Spans all table partitions
  • Eventually consistent reads only
  • Has own capacity settings
  • Up to 20 GSIs per table

Local Secondary Index (LSI)

  • Same partition key as base table
  • Different sort key
  • Strongly consistent reads possible
  • Shares capacity with base table
  • 10GB limit per partition
Best Practice: Design indexes based on access patterns. Each GSI should serve specific query requirements to justify the additional cost.

Example: Football League Table

Base Table: Player data with UserId (PK) + GameTitle (SK) GSI: PlayersByTeam - TeamId (PK) + PlayerName (SK) Use case: "Show all players for Manchester United"

Capacity Management & Pricing

On-Demand Mode

  • Pay-per-request pricing
  • Automatic scaling
  • No capacity planning
  • Good for unpredictable workloads
  • Scales to zero when idle

Use when: Traffic is unpredictable or intermittent

Provisioned Mode

  • Reserve read/write capacity
  • More cost-effective for predictable traffic
  • Auto Scaling available
  • Good for consistent workloads

Use when: Traffic is predictable and consistent

Capacity Units

Read Capacity Units (RCU)

1 RCU = 1 strongly consistent read/second for item ≤ 4KB
1 RCU = 2 eventually consistent reads/second for item ≤ 4KB

Write Capacity Units (WCU)

1 WCU = 1 write/second for item ≤ 1KB

Free Tier: 25 GB storage + 25 RCUs + 25 WCUs (200M requests/month) - Always available

Cost Optimization Strategies

Capacity Optimization

  • Choose appropriate capacity mode for workload
  • Use Auto Scaling for provisioned mode
  • Monitor capacity utilization with CloudWatch
  • Right-size provisioned capacity

Storage Optimization

  • Use TTL to automatically delete expired items
  • Consider DynamoDB Standard-IA for infrequent access
  • Implement data archival strategies
  • Compress large items when possible

Feature Optimization

  • Minimize number of global table regions
  • Optimize secondary index usage
  • Use appropriate backup retention periods
  • Monitor DynamoDB Streams usage

Query Optimization

  • Use Query instead of Scan operations
  • Implement pagination for large result sets
  • Use projection expressions to limit returned data
  • Leverage eventually consistent reads when possible
Cost Components: Read/write requests, data storage, data transfer, global tables, backups, and additional features like DAX

Design Best Practices

Data Modeling

  • Start with access patterns, not entities
  • Single table design where possible
  • Denormalize data for read performance
  • Use high-cardinality partition keys

Performance Optimization

  • Prefer Query over Scan
  • Use eventually consistent reads
  • Implement pagination for large results
  • Avoid hot partitions

Cost Optimization

  • Choose appropriate capacity mode
  • Use TTL for automatic deletion
  • Monitor capacity utilization
  • Minimize secondary indexes

Security & Operations

  • Implement least privilege access
  • Enable encryption at rest
  • Use VPC endpoints for private access
  • Set up CloudWatch monitoring
Golden Rule: Design your table structure around your access patterns, not your data relationships
